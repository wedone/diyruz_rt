# CC2530 固件一条龙开发流程（AI 套用版）

> 文档位置：`d:\VC\diyruz_rt\docs\CC2530固件一条龙流程.md`
> 目标读者：AI Agent / 开发者，用于在 CC2530 + ESP8266 + CCLoader WebUI 环境下完成"改码→编译→转 BIN→烧录→监控→再改码"的完整闭环
> 关键原则：所有路径、命令、API 端点均经实际验证，AI 可直接套用

---

## 1. 概述

### 1.1 一条龙方案是什么

把 CC2530 固件开发从"手动多工具切换"升级为"AI 自主循环"：

| 传统方式痛点 | 一条龙方案解决 |
|---|---|
| IAR 必须手动打开 IDE 点编译 | IarBuild.exe CLI 编译，脚本化 |
| HEX 要手动转 BIN | `hex2bin.py` 自动转换 + 256KB 填充 |
| 烧录要拔插 USB-TTL、换工具 | CCLoader WebUI，WiFi 一键烧录 |
| 监控要再拔插一次线 | WebUI 同一套接线，状态机自动切换 |
| 改一行代码要重走整套流程 | 全部脚本化，AI 可循环执行 |

### 1.2 适用场景

- CC2530 Z-Stack 固件开发与调试
- DIYRuZRT 4 路智能开关固件迭代
- 任何基于 CC2530 + MT_UART 串口日志的项目
- 需要快速验证固件修改是否生效

### 1.3 硬件拓扑

```
   ┌──────────────┐   USB    ┌──────────────┐   4 根杜邦线   ┌──────────────┐
   │   开发电脑   │──────────│ NodeMCU      │────────────────│  CC2530      │
   │ (PowerShell) │          │ ESP8266      │  RST/DC/DD/RX  │  模块        │
   └──────┬───────┘          │ IP:10.0.0.147│                └──────────────┘
          │                  └──────┬───────┘
          │ WiFi (HTTP API)         │ CC Debug + UART 监听
          └──────────────────────────┘
```

电脑不需要直连 CC2530，所有操作通过 ESP8266 的 HTTP API 完成。

### 2.3 接线核对

烧录与监控共用同一套接线，**全程无需拔插**：

| ESP8266 | CC2530 | 用途 |
|---|---|---|
| D1 (GPIO5)  | Pin 7 RESETn | CC Debug 复位 |
| D2 (GPIO4)  | Pin 3 DC      | CC Debug 时钟 |
| D6 (GPIO12) | Pin 4 DD      | CC Debug 数据 |
| RX (GPIO3)  | P0_3 UART0 TX | 串口监控（接收 CC2530 日志） |
| GND         | GND           | 共地（必须） |
| 3V3         | VCC           | 供电（< 50mA） |

> **重要**：P0_3 是 CC2530 的 UART0 TX（Alt-1 映射），ESP8266 的 GPIO3 是硬件 UART0 RX，直连即可。

## 3. 标准循环流程（7 步）

### 步骤 1：修改固件源码

按需编辑 `d:\VC\diyruz_rt\` 下的源文件。常见修改点：

- `DIYRuZRT/Source/preinclude.h` — 编译开关（调试串口、波特率、缓冲）
- `DIYRuZRT/Source/zcl_DIYRuZRT.c` — 应用层业务逻辑、日志
- `ZMain/TI2530DB/ZMain.c` — 启动流程
- `Components/mt/MT_UART.c` — MT_UART 初始化

> 关键配置见 [第 4 节：preinclude.h 调试开关](#4-preincludeh-调试开关)。

### 步骤 2：IAR CLI 编译

在 PowerShell 中执行（路径含空格必须加引号）：

```powershell
& "C:\Program Files (x86)\IAR Systems\Embedded Workbench 8.0\common\bin\IarBuild.exe" `
    "d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\DIYRuZRT.ewp" `
    -build RouterEB
```

**成功标志**：最后输出 `Total number of errors: 0`，且 `DIYRuZRT.hex` 文件更新时间刷新。

**常见错误**：
- `XDATA_I segment too large` → 缩小 `MT_UART_TX_BUFF_MAX`/`MT_UART_RX_BUFF_MAX`（见 preinclude.h）
- `File not found` → 检查 .ewp 路径
- `License error` → IAR 许可证问题，需打开 GUI 激活

### 步骤 3：HEX 转 BIN

CCLoader 要求 256KB 完整 BIN，必须转换：

```powershell
python d:\VC\diyruz_rt\Tools\hex2bin.py `
    d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\RouterEB\Exe\DIYRuZRT.hex `
    d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\RouterEB\Exe\DIYRuZRT.bin `
    --pad-to 0x40000
```

**成功标志**：输出 `已写入: ...DIYRuZRT.bin (262144 字节)`，BIN 大小必须是 256KB（262144 字节）。

> **简化方案**：CCLoader WebUI 浏览器端支持直接上传 .hex 文件，会自动转 BIN。但 AI 自动化场景下建议本地转换，避免依赖浏览器。

### 步骤 4：上传 BIN 到 ESP8266

通过 HTTP API 上传（PowerShell 用 `curl.exe`，不是 `curl` 别名）：

```powershell
curl.exe -s -F "file=@d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\RouterEB\Exe\DIYRuZRT.bin" `
         http://10.0.0.147/api/upload
```

**成功响应**：
```json
{"success":true,"filename":"DIYRuZRT.bin","size":262144}
```

### 步骤 5：异步烧录（推荐）

异步模式避免 HTTP 长连接超时，通过轮询 status 跟踪进度：

```powershell
# 发起异步烧录
$body = '{"filename":"DIYRuZRT.bin","verify":false}'
curl.exe -s -X POST "http://10.0.0.147/api/burn?async=1" `
         -H "Content-Type: application/json" `
         -d $body
```

**成功响应**（HTTP 202）：
```json
{"success":true,"async":true,"task_id":1,"total_blocks":512}
```

> **PowerShell JSON 转义陷阱**：简单 JSON 可用单引号包裹（如上），但复杂 JSON 建议写入临时文件后用 `-d "@file.json"`，避免转义问题。参考 `d:\VC\diyruz_rt\Tools\burn_req.json`。

### 步骤 6：轮询烧录进度

```powershell
# 每 2 秒查询一次，直到 done=true
while ($true) {
    $status = curl.exe -s http://10.0.0.147/api/status | ConvertFrom-Json
    $burn = $status.burn
    Write-Host "进度: $($burn.percent)% ($($burn.current_block)/$($burn.total_blocks))"
    if ($burn.done) {
        if ($burn.error) { Write-Host "失败: $($burn.error)"; break }
        Write-Host "烧录成功"; break
    }
    Start-Sleep -Seconds 2
}
```

**256KB BIN 烧录时间**：约 3 分钟（CC Debug 协议，2KB 块为单位）。

### 步骤 7：监控串口日志

```powershell
# 1. 启动监控（自动复位 CC2530，捕获完整启动日志）
$monBody = '{"baud":115200,"auto_reset":true}'
curl.exe -s -X POST http://10.0.0.147/api/monitor `
         -H "Content-Type: application/json" `
         -d $monBody

# 2. 轮询日志（断点续传，避免数据丢失）
$offset = 0
$deadline = (Get-Date).AddSeconds(15)
while ((Get-Date) -lt $deadline) {
    $resp = curl.exe -s "http://10.0.0.147/api/monitor/buffer?since=$offset" | ConvertFrom-Json
    if ($resp.data) {
        # Base64 解码
        $bytes = [System.Convert]::FromBase64String($resp.data)
        [System.Text.Encoding]::UTF8.GetString($bytes) | Write-Host -NoNewline
        $offset = $resp.total
    }
    Start-Sleep -Milliseconds 300
}

# 3. 停止监控
curl.exe -s -X POST http://10.0.0.147/api/stop
```

**成功标志**：收到 `[DIY] zclDIYRuZRT_Init done` 等业务日志，说明固件已正常运行。

> 监控缓冲为 8192 字节环形，启动日志很快可能丢失前面部分。如需完整启动日志，`auto_reset=true` 会在进入监控时主动复位 CC2530，配合立即轮询可捕获大部分日志。

### 步骤 8：循环回到步骤 1

根据监控日志分析固件行为，若需修改则回到步骤 1，重复直到固件符合预期。

## 4. preinclude.h 调试开关

`d:\VC\diyruz_rt\DIYRuZRT\Source\preinclude.h` 是固件调试的核心配置文件，控制 MT_UART 串口输出。

### 4.1 标准调试配置（推荐）

```c
// ====== 调试串口开关 ======
#define ZTOOL_P1
#define DIY_DEBUG_UART

#if defined(DIY_DEBUG_UART)
#define MT_UART_TX_BUFF_MAX  64
#define MT_UART_RX_BUFF_MAX  64
#define MT_UART_DEFAULT_BAUDRATE  HAL_UART_BR_115200
// 关键：禁用硬件流控，否则 CTS 悬空导致 UART 卡死
#define MT_UART_DEFAULT_OVERFLOW  FALSE
#endif
```

### 4.2 各开关含义

| 开关 | 作用 | 注意事项 |
|---|---|---|
| `ZTOOL_P1` | 启用 MT_UART，让 Z-Stack 通过 UART0 输出系统日志 | 启用后 P0_2/P0_3 让给 UART，LED3/LED4 失效 |
| `DIY_DEBUG_UART` | 缩小 UART DMA 缓冲（节省 XDATA）+ 业务日志开关 | 关闭后业务日志宏 `DIY_LOG(s)` 变空操作 |
| `MT_UART_TX_BUFF_MAX` | TX 缓冲字节数 | 默认 128，调试日志单行 < 50 字节，64 足够 |
| `MT_UART_RX_BUFF_MAX` | RX 缓冲字节数 | 同上 |
| `MT_UART_DEFAULT_BAUDRATE` | MT_UART 波特率 | CC2530 @ 32MHz 下 `HAL_UART_BR_115200` 对应 U0GCR=11/U0BAUD=216 |
| `MT_UART_DEFAULT_OVERFLOW` | 硬件流控（RTS/CTS）开关 | **必须 FALSE**！CC2530 模块 CTS 悬空会导致发送卡死 |

### 4.3 关闭调试模式（发布版）

```c
// 注释掉这两行即可恢复 LED3/LED4，关闭 MT_UART 输出
// #define ZTOOL_P1
// #define DIY_DEBUG_UART
```

### 4.4 历史教训：硬件流控卡死

**症状**：固件烧录后串口完全无输出（0 字节）。

**定位过程**：
1. 在 `ZMain.c` main() 各阶段加 `M1-M9` 裸寄存器 UART 诊断
2. 发现 M1-M7 丢失，仅 M8 后能收到 → 卡死在 `osal_init_system()`
3. 在 `OSAL_DIYRuZRT.c` 的 `osalInitTasks()` 加 `T1-T13` 细粒度诊断
4. 定位到 `MT_TaskInit()` 内部的 `MT_UartInit()` 卡死
5. 再加 `U-V-W` 和 `A-F` 诊断，确认卡在 `HalUARTOpenDMA()` 的 `if (config->flowControl)` 分支

**根本原因**：`MT_UART.h` 第 71 行 `#define MT_UART_DEFAULT_OVERFLOW TRUE` 默认启用硬件流控。CC2530 模块的 CTS 引脚悬空（高电平），UART 认为接收方未就绪，发送 DMA 永远等待。

**修复**：在 `preinclude.h` 显式定义 `MT_UART_DEFAULT_OVERFLOW FALSE`。

## 5. 业务日志规范

### 5.1 日志输出方式

业务代码使用 `HalUARTWrite(HAL_UART_PORT_0, ...)` 输出日志，封装在 `zcl_DIYRuZRT.c` 中：

```c
#if defined(DIY_DEBUG_UART)
#include "hal_uart.h"
#define DIY_LOG_PORT  HAL_UART_PORT_0
static void diy_log_str(const char *s);        // 输出字符串
static void diy_log_u8_dec(uint8 v);           // 输出十进制
static void diy_log_u8_hex(uint8 v);           // 输出十六进制
static void diy_log_line(const char *s);       // 输出字符串 + \r\n
#define DIY_LOG(s)          diy_log_line(s)
#define DIY_LOG_HEX(v)      diy_log_u8_hex(v)
#else
#define DIY_LOG(s)
#define DIY_LOG_HEX(v)
#endif
```

### 5.2 已有日志点

- `zclDIYRuZRT_Init` 完成：`[DIY] zclDIYRuZRT_Init done`
- 继电器状态：`[DIY] relay_state=0x00`
- Commissioning 状态：`[DIY] commissioning mode=0 status=2`
- 按键事件、继电器控制等

### 5.3 添加新日志

在业务代码中直接调用 `DIY_LOG("...")` 即可，关闭 `DIY_DEBUG_UART` 后自动变空操作，不影响发布版。

## 6. 完整自动化脚本（PowerShell）

把步骤 2-7 合并为一个脚本，AI 可直接调用：

```powershell
# burn_and_monitor.ps1 - CC2530 一条龙烧录监控
# 用法：powershell -File burn_and_monitor.ps1
# 前置：已修改源码，IAR 已激活

$ErrorActionPreference = "Stop"

$IAR = "C:\Program Files (x86)\IAR Systems\Embedded Workbench 8.0\common\bin\IarBuild.exe"
$EWP = "d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\DIYRuZRT.ewp"
$CFG = "RouterEB"
$HEX = "d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\RouterEB\Exe\DIYRuZRT.hex"
$BIN = "d:\VC\diyruz_rt\DIYRuZRT\CC2530DB\RouterEB\Exe\DIYRuZRT.bin"
$HEX2BIN = "d:\VC\diyruz_rt\Tools\hex2bin.py"
$IP = "10.0.0.147"
$MONITOR_SEC = 15

Write-Host "=== 步骤 1/6: IAR 编译 ==="
& $IAR $EWP -build $CFG
if ($LASTEXITCODE -ne 0) { throw "IAR 编译失败" }

Write-Host "=== 步骤 2/6: HEX 转 BIN ==="
python $HEX2BIN $HEX $BIN --pad-to 0x40000
if ($LASTEXITCODE -ne 0) { throw "HEX 转 BIN 失败" }

Write-Host "=== 步骤 3/6: 上传 BIN ==="
$up = curl.exe -s -F "file=@$BIN" "http://$IP/api/upload" | ConvertFrom-Json
if (-not $up.success) { throw "上传失败: $up" }
Write-Host "上传成功: $($up.filename) ($($up.size) 字节)"

Write-Host "=== 步骤 4/6: 异步烧录 ==="
$body = '{"filename":"DIYRuZRT.bin","verify":false}'
$burn = curl.exe -s -X POST "http://$IP/api/burn?async=1" `
    -H "Content-Type: application/json" -d $body | ConvertFrom-Json
if (-not $burn.success) { throw "发起烧录失败: $burn" }
Write-Host "task_id=$($burn.task_id) blocks=$($burn.total_blocks)"

Write-Host "=== 步骤 5/6: 轮询进度 ==="
while ($true) {
    $st = curl.exe -s "http://$IP/api/status" | ConvertFrom-Json
    $b = $st.burn
    Write-Host "`r进度: $($b.percent)% ($($b.current_block)/$($b.total_blocks))" -NoNewline
    if ($b.done) {
        Write-Host ""
        if ($b.error) { throw "烧录失败: $($b.error)" }
        Write-Host "烧录成功"
        break
    }
    Start-Sleep -Seconds 2
}

Write-Host "=== 步骤 6/6: 监控串口（${MONITOR_SEC}秒）==="
$mon = curl.exe -s -X POST "http://$IP/api/monitor" `
    -H "Content-Type: application/json" `
    -d '{"baud":115200,"auto_reset":true}'
Write-Host "监控已启动"

$offset = 0
$deadline = (Get-Date).AddSeconds($MONITOR_SEC)
while ((Get-Date) -lt $deadline) {
    $resp = curl.exe -s "http://$IP/api/monitor/buffer?since=$offset" | ConvertFrom-Json
    if ($resp.data) {
        $bytes = [System.Convert]::FromBase64String($resp.data)
        [System.Text.Encoding]::UTF8.GetString($bytes) | Write-Host -NoNewline
        $offset = $resp.total
    }
    Start-Sleep -Milliseconds 300
}

curl.exe -s -X POST "http://$IP/api/stop" | Out-Null
Write-Host "`n=== 完成 ==="
```

## 7. CCLoader WebUI API 速查

完整文档见 `D:\VC\CCLoader\README.md` 第 7 章，此处仅列 AI 自动化常用端点。

### 7.1 烧录相关

| 方法 | 路径 | 用途 |
|---|---|---|
| GET  | `/api/status` | 查询状态（state/burn/monitor/wifi） |
| POST | `/api/upload` | 上传 BIN（multipart, 字段 `file`） |
| POST | `/api/burn?async=1` | 异步烧录（立即返回 task_id） |
| POST | `/api/burn` | 同步烧录（阻塞至完成，不推荐） |
| DELETE | `/api/files/{name}` | 删除已上传 BIN |

### 7.2 监控相关

| 方法 | 路径 | 用途 |
|---|---|---|
| POST | `/api/monitor` | 开始监控（body: `{"baud":115200,"auto_reset":true}`） |
| POST | `/api/stop` | 停止监控 |
| POST | `/api/reset` | 复位 CC2530（GPIO5 拉低 RESETn） |
| GET  | `/api/monitor/buffer` | 获取全部监控缓冲 |
| GET  | `/api/monitor/buffer?since=N` | 断点续传（从累计字节 N 之后） |

### 7.3 状态机互斥

| 当前状态 | 允许 | 拒绝 |
|---|---|---|
| `idle` | 所有 | - |
| `burning` | `/api/status` | `/api/burn`、`/api/monitor`、`/api/reset` 返回 409 |
| `monitoring` | `/api/status`、`/api/stop`、`/api/reset`、`/api/monitor/buffer` | `/api/burn`、`/api/monitor` 返回 409 |

### 7.4 监控响应字段

```json
{
  "total": 1234,
  "buffered": 500,
  "offset": 800,
  "data": "Base64编码的日志字节",
  "truncated": false,
  "missed": 0
}
```

- `total`：累计接收字节数（单调递增，作为下次 `since` 的值）
- `truncated=true`：缓冲溢出，旧数据被覆盖，`missed` 为丢失字节数
- 缓冲大小：8192 字节环形

## 8. 故障排查

### 8.1 编译阶段

| 现象 | 原因 | 解决 |
|---|---|---|
| `XDATA_I segment too large by 0xNN` | UART DMA 缓冲占用过多 XDATA | 在 preinclude.h 中缩小 `MT_UART_TX/RX_BUFF_MAX` 到 64 |
| `File not found: zcl_ha.c` | 路径配置错误 | 检查 .ewp 中的路径，参考 commit `e67f376` |
| IarBuild 找不到 | IAR 未安装或路径不同 | 搜索 `Get-ChildItem "C:\Program Files*\IAR Systems" -Filter IarBuild.exe -Recurse` |

### 8.2 烧录阶段

| 现象 | 原因 | 解决 |
|---|---|---|
| `chip not detected` | DD/DC/RESET 线接错或松动 | 检查 4 根线 + 共地 |
| `XOSC timeout` | CC2530 32MHz 晶振未起振 | 硬件问题，更换模块 |
| `verify failed at block N` | DD 线接触不良或 Flash 寿命尽 | 重新插紧；或换 CC2530 |
| 烧录卡 0% | CC2530 没供电或已锁死 | 检查 3V3 供电；按 RESET 后重试 |
| HTTP 超时 | 用了同步烧录 `/api/burn` | 改用 `?async=1` 异步模式 |

### 8.3 监控阶段

| 现象 | 原因 | 解决 |
|---|---|---|
| 监控 0 字节 | P0_3 → RX 接错 / 波特率不对 / 固件未配置 UART 输出 | 检查接线；尝试 9600/57600/115200；确认 `ZTOOL_P1` 已启用 |
| 监控有乱码 | 波特率不匹配 | 改波特率重试 |
| 监控卡顿丢数据 | WiFi 信号差 | RSSI 应 > -75dBm |
| 仅收到部分启动日志 | CCLoader 进入监控态前 CC2530 已启动 | `auto_reset=true` 会主动复位；接受不完整日志，靠后续业务日志判断 |
| Base64 解码失败 | PowerShell `[Convert]::FromBase64String` 严格 | 用 Python `base64.b64decode(data + '='*(-len(data)%4))` 补齐填充 |

### 8.4 串口完全无输出的诊断流程

如固件烧录后串口完全无输出，按以下顺序排查：

1. **确认接线**：P0_3（CC2530）→ RX/GPIO3（ESP8266），共地已接
2. **确认波特率**：CC2530 @ 32MHz 用 115200，尝试 9600/57600 兜底
3. **确认固件配置**：`preinclude.h` 中 `ZTOOL_P1` 和 `DIY_DEBUG_UART` 均已启用
4. **确认硬件流控**：`MT_UART_DEFAULT_OVERFLOW` 必须为 `FALSE`（最常见陷阱）
5. **加裸寄存器诊断**：在 `ZMain.c` main() 早期用直接操作 `U0DBUF` 的方式输出字符串，绕过 HAL 框架，定位卡死位置
6. **测电压**：万用表测 P0_3 引脚电压，发送时应有 ~3.3V 跳变

## 9. PowerShell 陷阱备忘

### 9.1 curl 别名问题

PowerShell 中 `curl` 是 `Invoke-WebRequest` 的别名，不支持 `-F`、`-d` 等参数。**必须用 `curl.exe`**：

```powershell
# 错误：会报参数不识别
curl -F "file=@x.bin" http://...

# 正确
curl.exe -s -F "file=@x.bin" http://...
```

### 9.2 命令分隔符

PowerShell 不支持 `&&`，用 `;` 分隔命令：

```powershell
# 错误
curl.exe ... && curl.exe ...

# 正确
curl.exe ... ; curl.exe ...
```

### 9.3 JSON 传参

简单 JSON 用单引号包裹即可，复杂 JSON 建议写临时文件：

```powershell
# 简单
curl.exe -d '{"filename":"x.bin","verify":false}' ...

# 复杂：写文件
$body | Out-File -FilePath req.json -Encoding ascii
curl.exe -d "@req.json" ...
```

### 9.4 HEREDOC 不支持

PowerShell 不支持 bash 的 `<<EOF` 语法，git commit 多行 message 用 `-F` 指定文件：

```powershell
# 写入临时文件
$msg | Out-File -FilePath .git\COMMIT_MSG.tmp -Encoding ascii
git commit -F .git\COMMIT_MSG.tmp
Remove-Item .git\COMMIT_MSG.tmp
```

## 12. 历史经验总结

| 教训 | 出处 | 避免方法 |
|---|---|---|
| `MT_UART_DEFAULT_OVERFLOW` 默认 TRUE 导致 UART 卡死 | 本次调试 | preinclude.h 显式定义为 FALSE |
| 关闭 ZTOOL_P1 后 P0_3 恢复为 LED4，无 UART 输出 | 本次调试 | 监控固件必须保留 ZTOOL_P1 |
| CCLoader 进入监控态有 ~100ms 延迟，丢失启动早期日志 | 本次调试 | 接受不完整日志，靠业务日志判断 |
| PowerShell `curl` 是别名，参数不兼容 | 本次调试 | 一律用 `curl.exe` |
| CCLib 烧录 256KB 需 ~3 小时，CCLoader 仅需 ~3 分钟 | 历史经验 | 用 CCLoader WebUI，不用 CCLib |
| CC2530 BIN 必须填充到 256KB (0x40000) | CCLoader 约束 | hex2bin.py 默认 `--pad-to 0x40000` |
| CC2530 UART0 Alt-1: P0_2=RX, P0_3=TX @ 32MHz | 硬件固定 | 接线时认准 P0_3 是 TX |
| 115200 @ 32MHz 对应 U0GCR=11, U0BAUD=216 | Z-Stack 配置 | 用 `HAL_UART_BR_115200` 宏，不要硬编码 |
