## 3. 根因分析

### 3.1 完整代码路径追踪

```
zclDIYRuZRT_Init()
  → s_init_reached=1
  → 注册 endpoint、按键、回调
  → s_init_reached=2
  → bdb_StartCommissioning(NWK_STEERING | FINDING_BINDING)
    → bdbCommissioningMode 设置 NWK_STEERING bit
    → bdb_startResumeCommissioningProcess()
      → NWK_STEERING 路径
      → bdb_nwkJoiningFormation(TRUE)  // bdb.c:1846
        → ZDOInitDeviceEx(100, 0)      // ZDApp.c:664
          → 设置 runtimeChannel
          → ZDApp_NetworkInit()         // ZDApp.c:3086
            → ZDO_NETWORK_INIT 事件
              → ZDO_StartDevice()       // ZDObject.c:280
                → 路由器 + MODE_JOIN 路径
                → NLME_NetworkDiscoveryRequest(runtimeChannel, scanDuration)  // ZDObject.c:322
                  → 应触发 MAC 层主动扫描（开启 RX 收 beacon）
                  → **但实际：FSMSTAT1 始终 0x00，MAC 层从未启动 RX**
  → s_init_reached=3
  → osal_start_reload_timer(EVT_RSSI, 100ms)
  → s_init_reached=4
```

### 3.2 MAC 层 RX 启动机制分析

CC2530 MAC 低层驱动路径（`Components/mac/low_level/srf04/`）：

```c
// mac_rx_onoff.c:204
void macRxOnRequest(void) {
  if (macRxEnableFlags) {     // ← 关键检查
    macRxOn();                // ← 实际启动 RX
  }
}

// mac_rx_onoff.c:253
static void macRxOn(void) {
  MAC_RADIO_RX_ON();          // 宏定义 = RFST = ISRXON
}
```

**关键发现**：`macRxOnRequest()` 只有 `macRxEnableFlags` 非零才调 `macRxOn()`。当前 `fsm=0x00` 说明：
1. 要么 `macRxEnableFlags` 为 0（MAC 层未请求 RX）
2. 要么 `macRxOn()` 被调用但 RF 状态机被外部因素干扰不响应

### 3.3 已排除的假设

| 假设 | 验证方式 | 结论 |
|---|---|---|
| DIY_LOG 被 MT 二进制帧吞掉 | 添加全局计数器 `s_init_reached` 等 | ❌ 假设错误，日志输出正常 |
| RF 硬件损坏 | 手动 `RFST=ISRXON` + 读 RSSI | ❌ RF 完全正常（fsm=0x15, RSSI=-87dBm） |
| 32MHz 晶振故障 | 检查 LOCK_STATUS 置位 | ❌ VCO 校准成功 |
| `mode=1` 是 Touchlink | 查 `bdb_interface.h` 枚举 | ❌ 是 NWK_STEERING 枚举值 |
| `status=2` 是 IN_PROGRESS | 查枚举 | ❌ 是 NO_NETWORK |
| Init 函数未执行完 | `init=4` 计数器 | ❌ 4 阶段全到达 |
| RFST 延迟不足导致 LOCK 未就绪 | 增加到 2ms 延迟 | ❌ 手动测试有效，但 MAC 层仍未启动 RX |
| NV 残留导致孤儿扫描 | `zgWriteStartupOptions(DEFAULT_NETWORK_STATE)` | ❌ 仍 fsm=0x00 |
| 加 INITIATOR_TL 会导致问题 | 移除后测试 | ✓ 确认会导致 MAC 不启动 RX，已移除 |
| 加 NWK_FORMATION 会导致问题 | 移除后测试 | ✓ 路由器不应创建网络，已移除 |

### 3.4 当前主怀疑方向：烧录器线缆干扰

**怀疑依据**：
1. CPU 能正常运行代码（init=4，日志输出正常）→ 排除完全复位
2. 手动发 ISRXON 后 RF 硬件完全正常（fsm=0x15, RSSI=-87dBm）→ RF 硬件本身没问题
3. 但 MAC 层从不启动 RX（fsm 始终=0x00）→ MAC 状态机被外部因素干扰
4. CCLoader（ESP8266）的 DD/DC/RESETN 线持续连接 CC2530

**可能机制**：
- CC2530 的 2-wire 调试接口（DD + DC + RESETN）在烧录器持续连接时可能锁住或干扰 MAC 内部状态机
- ESP8266 的 GPIO 可能泄漏电流或在复位时拉低 RESETN，导致芯片处于半复位状态
- 调试时钟线 DC 如果有噪声，可能误触发调试事件，干扰 AHB 总线对 RF 寄存器的访问
- Debug 模式可能阻止 MAC 层的某些状态转换

## 4. 建议下一步方案

### 方案 A（首选）：完全断开烧录器

1. 烧录完最新固件后，**拔掉烧录器所有线缆**（VCC/GND/DD/DC/RESETN 全部断开）
2. 用**独立 USB-TTL**（CH340/CP2102）连接：
   - USB-TTL 的 TX → CC2530 P0_2（UART0 RX）
   - USB-TTL 的 RX → CC2530 P0_3（UART0 TX）
   - USB-TTL 的 GND → CC2530 GND
   - USB-TTL 的 3.3V → CC2530 VCC（或独立 3.3V 供电）
3. 上电后观察 5 分钟，看 `fsm` 是否变为非 0 值，是否入网

**预期结果**：如果断开烧录器后 RF 能正常工作并入网，则问题彻底定位为烧录器干扰。

### 方案 B（如果方案 A 不便）：代码中持续手动启动 RX

修改 `zcl_DIYRuZRT.c` 的 RSSI 事件，每次都发 ISRXON：

```c
// 在 RSSI 事件处理中，读取前先手动启动 RX
RFST = 0xE3;  // ISRXON
volatile uint16 wait;
for (wait = 0; wait < 4000; wait++);  // ~2ms 等待 VCO 校准
// 然后再读 FSMSTAT1 和 RSSI
```

**注意**：这是治标不治本，方案 A 才能真正验证根因。且手动 ISRXON 不能替代 MAC 层主动扫描的时序控制，可能无法实际入网。

### 方案 C：深入排查 MAC 层 macRxEnableFlags

在 `mac_rx_onoff.c` 的 `macRxOnRequest()` 中添加日志，确认 `macRxEnableFlags` 是否为 0。需要 IAR 调试环境或修改协议栈源码。

### 方案 D：多方法并行测试

- 测试不同 PA/LNA 配置（CC2591/CC2592/SE2431L/无放大）
- 烧录 Koenkk 标准固件做对比测试

## 5. 调试工具与方法

### 5.1 诊断计数器机制

在 `zcl_DIYRuZRT.c` 中添加全局计数器，通过 ZDO_STATE_CHANGE 事件输出（避免与 MT 二进制帧冲突）：

```c
static uint8 s_init_reached = 0;     // Init 执行进度：0=未进入, 4=全部完成
static uint8 s_rssi_evt_count = 0;   // RSSI 事件触发次数
static uint8 s_zdo_change_count = 0; // ZDO_STATE_CHANGE 触发次数
```

输出格式：`init=X rssi=Y zdo=Z`

### 5.2 RF 寄存器直接读取

在 RSSI 事件中直接读取 CC2530 RF 寄存器（不发 RFST 指令，让 MAC 层自己管理）：

```c
fsm = (uint8)FSMSTAT1;       // bit0=RX_ACTIVE, bit1=TX_ACTIVE, bit2=LOCK_STATUS, bit4=CCA, bit5=SFD
freq = (uint8)FREQCTRL;      // 信道频率
valid = (uint8)(RSSISTAT & 0x01);  // RSSI_VALID
rssi_raw = (int8)RSSI;       // 8位有符号
rssi_dbm = (int16)rssi_raw - 61;  // dBm = RSSI + (-61)
```

### 5.3 监控工具链

| 工具 | 用途 |
|---|---|
| `Tools/hex2bin.py` | HEX→BIN 转换（填充 0xFF 到 256KB） |
| `Tools/read_monitor.py` | 启动监控（auto_reset=True），循环读取 buffer |
| CCLoader WebUI | ESP8266 烧录+监控一体机，HTTP API，IP=10.0.0.147 |

### 5.4 烧录器 API 工作流

```
upload → burn(async) → 轮询 /api/status → monitor(auto_reset) → 读取 buffer
```

注意：监控未停止就尝试烧录会返回 `{"error":"busy"}`，需先 `POST /api/stop`。
