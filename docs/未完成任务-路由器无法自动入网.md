## 2. 症状详情

### 2.1 串口日志模式

```
[DIY] zclDIYRuZRT_Init done
[DIY] relay_state=0x00
[DIY] commission mode=1 status=1         ← NWK_STEERING IN_PROGRESS
[DIY] ZDO_STATE_CHANGE: 2                ← DEV_NWK_ORPHAN（孤儿扫描）
[DIY] commission mode=1 status=1         ← NWK_STEERING IN_PROGRESS（回调通知）
[DIY] ZDO_STATE_CHANGE: 0                ← DEV_HOLD（扫描结束，未发现网络）
→ 循环重复（每轮约 2-3 秒）
```

**关键特征**：
- 仅出现 `mode=1`（NWK_STEERING），从未出现 `mode=0`（INITIALIZATION）或 `mode=2`（FORMATION）
- DEV_NWK_ORPHAN 到 DEV_HOLD 极快（< 1 秒），不像完整信道扫描（16 信道 × 120ms/信道 ≈ 1.9 秒）
- 无 beacon 发现迹象（没有 `DEV_NWK_DISC` 状态出现）

### 2.2 环境信息

| 项 | 值 |
|---|---|
| 路由器型号 | CC2530 模块（具体型号待确认） |
| 固件类型 | RouterEB（路由器） |
| 协调器 | 用户自供，permit join 4 分钟窗口 |
| 距离 | 紧挨着（无信号问题） |
| 信道配置 | `DEFAULT_CHANLIST=0x07FFF800`（全部 11-26 信道） |
| 扫描时长 | `BDB_DEFAULT_SCAN_DURATION=4`（≈ 120ms/信道） |

## 3. 根因分析（推测）

### 3.1 代码路径追踪

当前流程：
```
zclDIYRuZRT_Init()
  → zgWriteStartupOptions(ZCD_STARTOPT_DEFAULT_NETWORK_STATE)
  → bdb_StartCommissioning(NWK_FORMATION|NWK_STEERING|F&B)
    → bdb_initialization=TRUE
    → bdbNodeIsOnANetwork=FALSE（新设备）
    → bdbCommissioningMode 设置有 FORMATION+STEERING+F&B bits
    → bdb_startResumeCommissioningProcess()
      → NWK_STEERING 优先处理（bit 0）
        → Router 未入网 → bdb_nwkJoiningFormation(TRUE)
          → ZDOInitDeviceEx(100, 0)
            → DEV_NWK_ORPHAN → DEV_HOLD（极快失败）
      → NWK_STEERING 回调失败 → 清 mode → bdb_StartCommissioning(NWK_STEERING)
      → 循环（FORMATION 永远排不上，因为 NWK_STEERING 一直失败重试）
```

### 3.2 推测根因

`ZDOInitDeviceEx(100, 0)` 对路由器（未入网）的行为可能不是"全信道发现"而是"孤儿扫描"（orphan scan）：

1. `mode=0`→ 设置 centralized security（TC 地址=00:00:00:00:00:00:00:00）
2. `ZDApp_ReadNetworkRestoreState()` 新设备返回 `ZDO_INITDEV_RESTORED_NETWORK_STATE`
3. `ZDApp_RestoreNetworkState()` 恢复不存在的网络 → WEB 为 `NEW_NETWORK_STATE`
4. 但是 `ZDApp_NetworkInit(extendedDelay)` 的实际行为取决于 NV 残留

**可能原因**：
- NV 中有上次测试残留的网络参数，导致路由器认为"有过网络"而走孤儿扫描
- Router NWK_STEERING 的 ZDOInitDeviceEx 实现不走完整信道发现

### 3.3 已尝试但未生效的方案

| 尝试 | 结果 |
|---|---|
| `zgWriteStartupOptions(ZG_STARTUP_SET, ZCD_STARTOPT_DEFAULT_NETWORK_STATE)` | 无效，仍是孤儿扫描 |
| 全套 commission（FORMATION|STEERING|F&B|INITIATOR_TL） | NWK_STEERING 优先，FORMATION 排不上 |
| NWK_STEERING 失败后重试 | 循环重试但始终孤儿扫描 |
| INITIALIZATION 失败后重试 NWK_STEERING | 此回调路径实际上未被触发（bdb_reportCommissioningState 中状态不匹配） |

## 4. 建议下一步方案

### 方案 A（首选）：彻底清除 NV 重新开始

在烧录固件前使用 CCLoader 擦除整个 Flash，确保无 NV 残留：

```bash
# 使用 CCLoader WebUI 或 cc_write_flash.py 先擦除
# 然后烧录新固件
```

或在代码中显式调用 `bdb_setFN()` 后再启动 commissioning。

### 方案 B：绕过 NWK_STEERING 直接走 FORMATION

修改 `bdb_startResumeCommissioningProcess` 的优先级（但这是 TI 闭源代码不可改），或调整回调策略——在 BDB_INITIALIZATION 回调失败时使用 **FORMATION 而非 NWK_STEERING**。

但 `bdb_reportCommissioningState(BDB_INITIALIZATION, FALSE)` 不会发送通知（因为 `bdbCommissioningProcedureState.bdbCommissioningState` 不匹配），所以此回调路径不可用。

### 方案 C：使用长按 5 秒方式手动触发

当前长按 SW1 5 秒已经触发全套 commissioning。但需人工操作。

### 方案 D：深入调试 ZDApp 状态机

在 `ZDApp_NetworkInit()` 前后加更多串口日志，确认实际走的是孤儿扫描还是完整发现。需要 IAR 调试环境（CC Debugger）。

### 方案 E：尝试 End Device 模式

将编译配置改为 EndDeviceEB 而非 RouterEB，观察 End Device 的 NWK_STEERING 行为是否不同。
